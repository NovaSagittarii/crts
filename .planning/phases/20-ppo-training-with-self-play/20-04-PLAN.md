Create `packages/bot-harness/training/training-worker.ts` -- the worker_threads entry point.

CRITICAL: This file must import `@tensorflow/tfjs` (pure JS), NEVER `@tensorflow/tfjs-node`. The native addon crashes in worker threads (research Pitfall 1). If tfjs-node was the only installed package (tfjs pure JS not available), this file should import `@tensorflow/tfjs-node` ONLY if there's a single worker (no threading issue with 1 worker), otherwise throw an error explaining the limitation.

Architecture: Option B from research -- autonomous episode collection. Worker loads frozen weights into a local pure-JS model, runs full episodes, sends complete trajectories back.

Implementation:

1. Import `{ parentPort, workerData } from 'node:worker_threads'`.
2. Import `* as tf from '@tensorflow/tfjs'` (pure JS -- safe in workers).
3. Import BotEnvironment, RandomBot, NoOpBot from `#bot-harness` (these are pure TS, no native addons).
4. Import buildPPOModel, applyWeights from `./ppo-network.js` -- BUT the ppo-network module likely imports tfjs-node. To avoid this, the worker must build its model using `tf.model()` / `tf.layers.*` from the pure JS tfjs. So either:
   a. Duplicate the model-building logic with pure JS tf import (simpler, avoids cross-import issues), OR
   b. Make ppo-network.ts accept a `tf` module parameter so it can work with either backend.

   Preferred: option (a) -- create a `buildWorkerModel(config: PPOModelConfig)` function inside training-worker.ts that mirrors buildPPOModel but uses the pure JS import. This avoids coupling the worker to tfjs-node. Use the same layer names so weight shapes match exactly.

5. Define message protocol types:
   - `WorkerInitMessage`: `{ type: 'init', modelConfig: PPOModelConfig }`
   - `SetWeightsMessage`: `{ type: 'set-weights', weights: { shape: number[], buffer: ArrayBuffer }[] }`
   - `CollectEpisodeMessage`: `{ type: 'collect-episode', seed: number, opponentType: 'random' | 'noop' | 'checkpoint', opponentWeights: { shape: number[], buffer: ArrayBuffer }[] | null, episodeNumber: number }`
   - `EpisodeResultMessage`: `{ type: 'episode-result', trajectory: SerializedTrajectory, reward: number, ticks: number, won: boolean, workerId: number }`
   - `SerializedTrajectory`: `{ planes: ArrayBuffer[], scalars: ArrayBuffer[], actions: number[], rewards: number[], values: number[], logProbs: number[], dones: boolean[], actionMasks: ArrayBuffer[] }`

6. Worker lifecycle:
   a. On `'init'` message: build model from config using `buildWorkerModel()`, store reference.
   b. On `'set-weights'` message: apply weights to local model via applyWeights-style logic (create tensors from buffers, set on model, dispose temps).
   c. On `'collect-episode'` message:
   - Create BotEnvironment with grid config from workerData
   - Create opponent: RandomBot, NoOpBot, or a "checkpoint bot" that uses the opponent model weights (build a second local model, apply opponent weights, wrap in a BotStrategy adapter that calls the model's forward pass for each decideTick)
   - Set episode number on env
   - Call env.reset(seed, opponent)
   - Loop: forward pass through local model to get logits, sample masked action (apply mask before softmax), call env.step(action), record step (planes, scalars, action, reward, value from model, logProb, done, actionMask)
   - On termination: compute final value (0 if terminal, model value if truncated)
   - Serialize trajectory as plain ArrayBuffers
   - Post result back to main thread with transferable list for zero-copy

7. The "checkpoint bot" adapter: a class implementing BotStrategy that:
   - Holds a local tf.LayersModel with opponent checkpoint weights
   - In decideTick: encodes observation using ObservationEncoder, runs forward pass, samples from policy logits with action mask, decodes to BotAction via ActionDecoder
   - This allows checkpoint opponents to play in worker threads without tfjs-node

8. Handle `'terminate'` message: dispose model, call `process.exit(0)`.

Use `tf.tidy()` around all tensor operations in the episode loop to prevent leaks.

NOTE on testing (warning accepted): training-worker.ts has no dedicated unit test file. Its message protocol is exercised through the coordinator integration test (Task 2), which spawns real workers and verifies the full init -> set-weights -> collect-episode -> episode-result cycle. Mocking parentPort for isolated unit tests adds complexity without proportional value given the integration coverage.

1. `TrainingCoordinator` class -- the main thread orchestrator implementing D-14 actor-learner split:

   `async init(): Promise<void>`:
   a. Create BotEnvironment (for metadata only -- observationSpace, actionSpace)
   b. Build PPOModelConfig from env metadata and config.network

   d. Create PPOTrainer with model and config

   g. Create OpponentPool with config.selfPlay and logger.getCheckpointDir()
   h. Create TrainingLogger and call logger.init()
   i. Log config via logger.logConfig(config)
   j. If config.resumeRunId: load latest checkpoint (model weights + optimizer state) per D-11
   k. Spawn worker threads, send 'init' message with modelConfig to each

   `async run(): Promise<void>` -- main training loop:
   a. Extract current weights from model
   b. Send 'set-weights' to all workers per D-17
   c. For each collection round:
   - Sample opponents for each episode via opponentPool.sampleOpponent()
   - Distribute 'collect-episode' messages to workers (round-robin or queue-based)
   - If opponent is a checkpoint type, load weights and include in message
   - If opponent is built-in bot, send opponentType='random'/'noop'
   - Await all episode results from workers
     d. Deserialize trajectories into TrajectoryBuffer steps
     e. Finalize buffer (compute GAE with config.gamma, config.gaeLambda)
     f. Run PPO update: `trainer.update(buffer)`
     g. Log metrics: compute win rate from episode results, log via logger.logEpisode(), print live via logger.formatLiveMetrics()
     h. If opponentPool.shouldCheckpoint(episode): save checkpoint, add to pool
     i. Clear buffer, repeat from (a) until totalEpisodes reached
     j. Save final model to logger.getFinalModelDir()

   Worker management:
   - `spawnWorkers(count: number): Worker[]` -- creates Worker instances pointing to training-worker.ts, using `tsx` loader for TypeScript support if needed. Pass `workerData: { envConfig: { gridWidth, gridHeight, maxTicks } }`.
   - `broadcastWeights(workers: Worker[], weights: WeightData[]): void` -- sends 'set-weights' to each worker. Clone buffers for each worker (buffers can only be transferred once) per D-17.
   - `collectBatch(workers: Worker[], batchSize: number): Promise<EpisodeResult[]>` -- dispatches episodes to workers, collects results. Uses a promise-based queue: for each episode, pick the next idle worker, send 'collect-episode', await response.
   - `terminateWorkers(): Promise<void>` -- sends 'terminate' to all, awaits worker.terminate().

   Resume support (D-11):
   - `async saveOptimizerState(dir: string): Promise<void>` -- save optimizer weights to `optimizer-state.bin` alongside model checkpoint
   - `async loadOptimizerState(dir: string): Promise<void>` -- load and restore optimizer state from checkpoint directory
   - On resume: find latest checkpoint in existing run dir, load model weights + optimizer state, set episode counter

   `async cleanup(): Promise<void>`: terminate all workers, dispose model.
