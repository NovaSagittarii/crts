    Create `rating-pool.ts` implementing pool-based Glicko-2 rating management.

    **Class: RatingPool**

    Constructor takes `RatingPoolConfig` and optional `tau` (default from GLICKO2_DEFAULTS).

    State: `entities: Map<string, Glicko2Rating>` -- maps entity IDs to current ratings. `entityMatchCounts: Map<string, number>` -- match count per entity. `totalMatchesProcessed: number`.

    Methods:
    - `registerEntity(id: string): void` -- initializes entity with default Glicko-2 rating if not already registered.
    - `addEncounters(encounters: TemplateEncounter[]): void` -- collects encounters for the current period. Stores internally. Registers entities as discovered.
    - `runUpdate(): void` -- batch Glicko-2 update. For each entity:
      1. Collect all encounters where this entity appears (as entityA or entityB)
      2. Build MatchResult[] from opponent pre-update ratings and encounter scores
      3. Call updateRating with current entity rating and accumulated matches
      4. Store new rating (do NOT use new rating for other entities in same batch -- use pre-update snapshot)
      5. Increment matchCount for each entity by number of encounters processed
    - `getRatedEntities(): RatedEntity[]` -- returns sorted array of RatedEntity objects with provisional flag (RD > 150), pickRate (matchCount / totalMatchesProcessed), and empty outlierFlags.
    - `processMatches(matches: ParsedMatch[]): RatedEntity[]` -- convenience method: extracts encounters for this pool's game phase, registers entities, runs update, returns rated entities. Uses extractTemplateEncounters with tickRange from pool config.

    **CRITICAL batch update semantics (per research anti-pattern):** Before running updates, snapshot ALL entity ratings. During the update loop, use the SNAPSHOT for opponent ratings, not the updated values. After all entities are updated, replace the entity map with the new ratings.

    **Function: createRatingPools**

    `createRatingPools(options?: { perPhaseCombos?: boolean; tau?: number }): RatingPool[]`

    Creates the standard set of pools per D-02 and D-09:
    - 3 individual pools: individual-early (ticks 0-200), individual-mid (200-600), individual-late (600+)
    - 1 pairwise pool: pairwise-full (no tick filter) -- or 3 if perPhaseCombos
    - 1 frequent-set pool: frequent-set-full (no tick filter) -- or 3 if perPhaseCombos

    Default 5 pools. With `perPhaseCombos: true`, 9 pools.

    Use GAME_PHASE_DEFAULTS for tick boundaries. Use explicit `.js` extensions. Strict TypeScript.

  <action>
    **Part A: Create `combination-miner.ts`**

    Export `minePairwiseCombinations(matches: ParsedMatch[], tickRange?: { start: number; end: number }): Map<number, Map<string, number>>`:
    - For each match and team, extract the set of distinct templates used (within tickRange if provided)
    - For each team's template set, generate all 2-element subsets
    - Pair ID = sorted alphabetically joined with "+" (e.g., "block+glider")
    - Return Map<matchIndex, Map<pairId, minBuildCount>> where minBuildCount = min(countA, countB) among the pair's member templates. This is used by extractCombinationEncounters for log-weighting per D-08.

    Export `mineFrequentSets(matches: ParsedMatch[], options?: { minSupport?: number; maxSetSize?: number; tickRange?: { start: number; end: number } }): Array<{ setId: string; members: string[]; support: number }>`:
    - Per D-07 (frequent-set mining) with Claude's discretion on algorithm: Use direct enumeration (brute-force).
    - With only 5 templates, total possible subsets = 2^5 - 1 = 31. Enumerate all k-subsets for k=2..maxSetSize (default 4).
    - For each candidate subset, count support = number of (match, team) pairs where ALL members of the subset were used.
    - Filter by minSupport (default 5).
    - Set ID = sorted members joined with "+" (e.g., "block+generator+glider").
    - Default: minSupport=5, maxSetSize=4.

    **Part B: Create `outlier-detector.ts`**

    Export `detectOutliers(entities: RatedEntity[], options?: { sdThreshold?: number }): RatedEntity[]`:
    - Returns the same entities array with `outlierFlags` populated per D-10, D-11, D-12.
    - Does NOT mutate input -- creates new RatedEntity objects with updated outlierFlags.

    **Method A (D-10a) -- Statistical deviation:**
    1. Filter to non-provisional entities (RD <= 150)
    2. Compute mean and stddev of their ratings using existing `mean()` and `stddev()` from stats.ts
    3. sdThreshold defaults to 2.0 (per D-10: ">2 standard deviations")
    4. Flag entities with rating > mean + sdThreshold*SD as `statistical-outlier-high`
    5. Flag entities with rating < mean - sdThreshold*SD as `statistical-outlier-low`
    6. Provisional entities are NOT flagged by this method (insufficient data)

    **Method B (D-10b) -- Rating + usage matrix:**
    1. Compute median rating and median pick rate across non-provisional entities
    2. High rating = above median rating; High pick rate = above median pick rate
    3. Categorize: dominant (high+high), niche-strong (high+low), trap (low+high)
    4. Low rating + low pick rate = no flag (weak and unused)
    5. Provisional entities CAN receive usage-matrix flags if they have enough pickRate data

    Both methods are additive per D-12: an entity can be both `statistical-outlier-high` AND `dominant`.

    Use explicit `.js` extensions. Import mean/stddev from `./stats.js`. Strict TypeScript.
  </action>
