    **Part A: Create `rating-worker.ts`** -- worker thread entry point for parallel pool computation per D-05.

    Follow Phase 20's established pattern (training-worker.ts):
    - Import parentPort from 'node:worker_threads'
    - Message protocol:
      - Receive: `{ type: 'compute-pool'; poolConfig: RatingPoolConfig; entities: Array<{ id: string; rating: Glicko2Rating }>; encounters: TemplateEncounter[]; tau: number }`
      - Return: `{ type: 'pool-result'; poolName: string; entities: Array<{ id: string; rating: Glicko2Rating; matchCount: number }> }`
    - Worker creates a RatingPool from config, loads entities and encounters, runs batch update, returns results
    - Convert Maps to plain objects/arrays before postMessage (research Pitfall 5: Maps cannot be serialized)
    - NO TF.js import -- this is pure math only (research anti-pattern)
    - Use explicit `.js` extensions

    **Part B: Create `rating-coordinator.ts`** -- coordinates parallel pool computation per D-05.

    Export `computeRatingsSequential(matches: ParsedMatch[], options: RatingComputeOptions): Promise<RatingsReport>`:
    - Options: `{ tau?: number; perPhaseCombos?: boolean; sdThreshold?: number; minSupport?: number; maxSetSize?: number }`
    - Creates rating pools via createRatingPools
    - For individual pools: calls extractTemplateEncounters per match with pool's tick range
    - For pairwise pools: calls minePairwiseCombinations, then extractCombinationEncounters
    - For frequent-set pools: calls mineFrequentSets to discover sets, then rates them via encounters
    - Runs RatingPool.processMatches (or addEncounters + runUpdate) for each pool sequentially
    - Runs detectOutliers on each pool's results
    - Assembles RatingsReport from all pool results
    - Computes hyperparameters section from GLICKO2_DEFAULTS and GAME_PHASE_DEFAULTS

    Export `computeRatingsParallel(matches: ParsedMatch[], options: RatingComputeOptions & { workers?: number }): Promise<RatingsReport>`:
    - Same logic but dispatches pool computation to worker threads per D-05a
    - Uses _worker-shim.mjs for tsx TypeScript loading (same as Phase 20)
    - Worker count defaults to `min(poolCount, os.cpus().length - 1, 4)` -- leave 1 CPU for main thread
    - Spawns workers, sends serialized pool data, collects results via postMessage
    - Falls back to computeRatingsSequential if workers=1 or if pool count <= 2 (overhead not worth it)
    - Runs outlier detection in main thread AFTER all workers complete (needs full result set for SD computation)

    Export type `RatingComputeOptions` for consumer use.

    **Implementation notes:**
    - For frequent-set pools: first mine frequent sets from matches, then create encounters for each set entity. A frequent set entity participates in encounters like an individual template, but its weight is based on the minimum member build count (per D-08).
    - Outlier detection runs per pool (per-phase outliers, D-11) AND on a combined set (overall outliers).
    - Use path.join and import.meta.url/fileURLToPath for worker path resolution (follow training-coordinator.ts pattern).
