<objective>
Add the "Add Bot" capability to the socket contract, server, and web UI. The host can invite a bot to fill an empty slot in the lobby. The server validates the request and marks the slot as awaiting a bot. The web client shows an "Add Bot" button for empty slots (host only) and displays a bot indicator badge for bot players.

Purpose: This is the protocol-level foundation that enables the bot process (Plan 3) to join rooms. Per D-01, the host controls when/if a bot joins. Per D-03, bots are visually distinguishable.

Output: Updated socket contract with bot:add event, server handler, and web UI additions.
</objective>

    4. Add `'bot:add'` to `ClientToServerEvents`:


    **server.ts changes:**

    1. Add a `Set<string>` called `botSessionIds` at the module level (near the rooms Map) to track which sessionIds are bots. This set is checked when building membership payloads.

    2. Add `bot:add` socket event handler inside `io.on('connection', ...)`:
       - Validate sender is host: `room.lobby.snapshot().hostSessionId === session.id`, else emit `room:error` with `'not-host'`.
       - Validate room status is `'lobby'`: `room.status !== 'lobby'` => error `'not-in-lobby'`.
       - Validate slotId: Check `payload.slotId` is a string, exists in `room.lobby.snapshot().slotDefinitions`, else error `'invalid-slot'`.
       - Validate slot has capacity: Check the slot has fewer members than capacity.
       - Generate a `botSessionId` like `bot-${crypto.randomUUID().slice(0, 8)}`. Add to `botSessionIds` set.
       - Emit `'bot:added'` back to the requesting socket with `{ roomId, slotId, botSessionId }`.
       - The bot:added response tells the host the sessionId the bot process should use when connecting, so the host can display "Bot invited" status. The bot process itself will connect with this sessionId via CLI flag.
       - Call `emitMembership(room)` to update all clients.

    3. Update the membership payload builder to include `isBot: botSessionIds.has(participant.sessionId)` on each `MembershipParticipant`. Find where `MembershipParticipant` objects are constructed (in the `buildMembershipPayload` or equivalent function) and add the `isBot` field.

    4. When a bot session connects (sessionId starts with `bot-` and is in `botSessionIds`), set `session.name` to `'Bot'` (the bot process will set its own display name via `player:set-name` to something like "Bot (PPO-v3)" per D-03).

    Do NOT modify the existing `room:join`, `room:claim-slot`, or `room:set-ready` handlers -- the bot process will use them identically to a human client per D-02.
