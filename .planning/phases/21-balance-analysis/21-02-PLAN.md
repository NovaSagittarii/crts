  <action>
    Create `packages/bot-harness/analysis/win-rate-analyzer.ts` with two main exported functions:

    **`computeTemplateWinRates(matches: ParsedMatch[], config: AnalysisConfig): TemplateWinRate[]`** (per D-01):

    For each match, determine the winning teamId (or isDraw for 0.5 attribution). Extract all build actions with `result === 'applied'` and valid `templateId`. Get template names from `createDefaultStructureTemplates()`.

    Three attribution methods:
    1. **Presence-based** (D-01a): For each template, count matches where it was built by the winning team (1.0 for win, 0.5 for draw). Total = matches where template appeared for that team.
    2. **Usage-weighted** (D-01b): Weight each match contribution by how many times the template was built. A team building "block" 5 times in a win contributes 5.0; in a draw, 2.5.
    3. **First-build** (D-01c): Only consider the first `config.firstNBuilds` builds per team per match. Apply presence-based logic on this subset.

    For each attribution method, compute Wilson score CI using `wilsonScoreInterval(wins, total, config.confidence)`.

    **`computeStrategyWinRates(matches: ParsedMatch[], assignments: StrategyAssignment[], config: AnalysisConfig): StrategyWinRate[]`** (per D-03):

    Group matches by strategy label (from assignments). For each strategy, compute the same three attribution methods treating strategy presence as "did this team use this strategy in this match."

    Implementation details:
    - Draws count as 0.5 wins per D-02 research recommendation (standard in game balance).
    - Skip build actions with missing templateId (graceful degradation for old logs). Count skipped actions in metadata.
    - Only count `result === 'applied'` builds (rejected builds don't affect game state).
    - Use explicit `.js` extensions in imports. Explicit return types on all exports.
  </action>
