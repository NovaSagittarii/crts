<objective>
Restructure the training coordinator's synchronous generation loop into a pipelined design where episode collection overlaps with PPO gradient updates, I/O operations are non-blocking, and pipeline performance is measured per generation.

Purpose: Eliminate the idle gap where workers wait for the main thread to finish PPO updates, GAE computation, logging, and checkpointing. This is the single largest throughput bottleneck in the training pipeline.

Output: Modified `training-coordinator.ts` with pipelined `run()` loop, `PipelineMetrics` type, and `episodesPerSec` in `TrainingProgressData`.
</objective>

Add a private helper method `computePipelineMetrics` to TrainingCoordinator:

```typescript
private computePipelineMetrics(
  genStartMs: number,
  genEndMs: number,
  collectMs: number,
  processMs: number,
  episodesInBatch: number,
): PipelineMetrics {
  const wallMs = genEndMs - genStartMs;
  const overlapMs = Math.max(0, collectMs + processMs - wallMs);
  return {
    generationWallMs: wallMs,
    collectWallMs: collectMs,
    processWallMs: processMs,
    overlapMs,
    episodesPerSec: wallMs > 0 ? (episodesInBatch / wallMs) * 1000 : 0,
    pipelineEfficiency: Math.min(collectMs, processMs) > 0
      ? overlapMs / Math.min(collectMs, processMs)
      : 0,
  };
}
```

Restructure the `run()` method into a pipelined design. The new flow:

**Bootstrap (first generation):** Fully synchronous -- extract weights, broadcast, collect first batch. This establishes the pipeline.

**Steady-state loop:**
1. Start next batch collection as a detached Promise (workers still have current-gen weights -- stale-by-one is acceptable per CONTEXT.md). Attach `.catch()` immediately to prevent unhandled rejection.
2. Process current batch on main thread: deserialize trajectories into TrajectoryBuffer, finalize GAE, run PPO update, log metrics, fire onProgress callbacks.
3. Fire-and-forget checkpointing: if checkpoint due, start as a Promise but do NOT await yet.
4. Extract new weights from the updated model.
5. Await the next batch Promise (workers are done or finishing).
6. Broadcast new weights to all workers (they are idle now, having finished collecting).
7. Await any pending checkpoint I/O.
8. Clear the current batch's buffer.
9. Increment generationCounter.
10. Loop back to step 1.

**Critical invariants to maintain:**
- Workers receive weights from generation N before collecting episodes for generation N+1. The pipeline ensures this: `broadcastWeights(N)` completes before `collectBatch(N+1)` begins. The overlap is between `collectBatch(N+1)` and `processCurrentBatch(N)`, NOT between `collectBatch(N+1)` and `broadcastWeights(N)`.
- Actually, in the pipelined version, workers collect the NEXT batch with the CURRENT generation's weights (one gen stale), which is explicitly acceptable. The sequence is: after processing batch N, we extract new weights (gen N+1), await the already-running next batch (collected with gen N weights), then broadcast gen N+1 weights. So the NEXT next batch will use gen N+1 weights. Maximum staleness = 1 generation.
- `onProgress` callbacks fire only during current-batch processing, never from a background batch. Episode numbering remains monotonically increasing.
- Pause check: check `this.paused` and `this.stopRequested` BOTH at loop top AND after awaiting the next batch. If stop requested during background collection, await the background batch (prevent dangling Promise) then break.
- Fire-and-forget I/O: checkpoint save and logging run as Promises collected into a `pendingIO` array. Await them at a safe synchronization point (after next batch await, before starting the next processing cycle).

**Timing instrumentation:**
- Use `performance.now()` (from `node:perf_hooks`) for all timing measurements.
- Track `collectStartMs`/`collectEndMs` for the background batch collection.
- Track `processStartMs`/`processEndMs` for main-thread processing.
- Compute `PipelineMetrics` at end of each generation via `computePipelineMetrics()`.
- Include `episodesPerSec` in the onProgress callback data.

**Backward compatibility:**
- `onProgress` fires per-episode with the same `TrainingProgressData` shape, plus the new `episodesPerSec` field.
- `togglePause()` and `requestStop()` continue to work -- pause is checked at loop top and after background await.
- `generationStartTime` in onProgress data uses `performance.now()` converted to wall-clock (or just `Date.now()` at the start of the generation's processing phase, NOT the collection phase, per research recommendation).
- The final model save at end of `run()` remains unchanged.
- All existing public methods (`getEpisodeCounter`, `getRunId`, `getWinRate`, `getLogger`) remain unchanged.

**Error handling for background Promise:**

Change `await this.logger.logEpisode(entry)` to push the Promise into `pendingIO` array instead of awaiting inline. Await `Promise.all(pendingIO)` after the next batch is awaited. Same for checkpoint save.

Do NOT modify any other methods on TrainingCoordinator (init, cleanup, spawnWorkers, broadcastWeights, collectBatch, runWorkerQueue, terminateWorkers, resume support, deserializeTrajectory). Only the `run()` method body changes, plus the new private `computePipelineMetrics` helper.
